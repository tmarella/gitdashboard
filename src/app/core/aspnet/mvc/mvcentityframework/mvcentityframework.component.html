Entity Framework

Entity framework is object relational mapping framework that offers automated mechanism for storing and accessing data in database.

Advantages
  • Provides auto generated code
  • Reduce development time and development cost
  • Enables developers to visually design models and mapping of database
  • Itâ€™s easy to map business objects (with drag & drop tables).
  • Allows customized mappings like single entity to multiple tables and even multiple entities to single table
Disadvantages
  • Lazy loading is the main drawbacks of EF
  • Its syntax is complicated
  • It is not available for every RDMS
  • It does not work if we change any schema of the database. We need to update the schema on the solution.
  • It is not good for huge domain model.
Code First Approach

This approach starts with writing code in c# and then EF will create database from your code.

  • Install Entity framework nuget package
  • Add database connection string information in config file
  • Add a POCO class
  • {{poco}}
  • Create another class and inherit DBContext
  • Declare a DBSet
  • {{dbset}}
  • Perform DBContext operations
  • Execute command 'Enable migrations' in package manager console. This is required only once for the project.
  • Execute 'Add migrations <migration-name>' in package manager console
  • Execute 'Update-Database' in package manager console
  • Verify database in SSMS
Database First Approach

This approach starts with creating database and tables and then EF will create domain classes.

  • Create Database and table in SSMS
  • In VS, add a new ADO.NET entity data model and select EF Designer from database
  • Select existing database, select framework and select tables, views and stored procedures
  • Entity model(.edmx) and POCO classes are generated in VS
  • Create an instance for DBContext class usually <dbname+entities>
  • Create a new object of poco class
  • Save changes to the database using SaveChanges()
  • Run the application
  • Verify changes in database in SSMS
Eager loading
  • Eager Loading helps you to load all your needed entities at once. i.e. related objects (child objects) are loaded automatically with its parent object.
  • Use Eager Loading when the relations are not too much. Thus, Eager Loading is a good practice to reduce further queries on the Server.
  • Use Eager Loading when you are sure that you will be using related entities with the main entity everywhere
Lazy loading
  • In case of lazy loading, related objects (child objects) are not loaded automatically with its parent object until they are requested. By default LINQ supports lazy loading.
  • Use Lazy Loading when you are using one-to-many collections.
  • Use Lazy Loading when you are sure that you are not using related entities instantly.
